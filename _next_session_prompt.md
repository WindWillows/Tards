## 下一阶段目标：引擎架构重构

**核心任务**：从"卡牌填充期"转入"引擎优化期"。

### 待办方向（与用户确认优先级后展开）

1. **EventBus 性能优化**
   - `*` 通配符监听器（如三氟化氯）每次事件都执行，高频事件（如 `card_moved`）下性能开销显著
   - 考虑：事件分类缓存、条件过滤提前化、或改为按需订阅模式

2. **费用系统统一**
   - 当前费用修正分散在 `player._cost_modifiers`、`card._card_cost_modifiers`、卡牌实例属性（如 `_shuixieshi_discount`）
   - 统一为 `CostModifierSystem`，支持叠加、来源追踪、自动清理

3. **关键词动态系统**
   - `_aura_keyword_fns` 是优秀的动态关键词机制，但缺乏统一注册/清理API
   - 抽象为 `AuraKeywordProvider`，支持死亡自动注销、多来源叠加

4. **指向系统清理**
   - 旧模式 `extra_targeting_stages` 与新模式 `TargetingRequest` 并存
   - 逐步迁移遗留卡牌，最终废弃 `extra_targeting_stages`

5. **监听器生命周期管理**
   - 当前 `owner=minion` 或 `owner=card` 的清理机制不一致
   - 卡牌弹回手牌、移除游戏时，监听器残留风险
   - 统一 `move_to()` 时的自动清理钩子

6. **卡牌池缓存**
   - `DEFAULT_REGISTRY.all_cards()` 每次调用都创建新列表
   - 为按标签/类型过滤添加缓存（如纯净异象池、时刻池）

### 紧急度评估

| 问题 | 影响范围 | 紧急度 |
|------|---------|--------|
| `*` 通配符性能 | 所有带实时aura的卡牌 | 高 |
| 监听器残留 | 弹回/移除类效果 | 高 |
| 费用系统统一 | 所有减费/增费卡牌 | 中 |
| 指向系统清理 | 遗留旧模式卡牌 | 中 |
| 关键词动态系统 | 条件关键词卡牌 | 低 |
| 卡牌池缓存 | 开发/检索类效果 | 低 |
