## 1. 本地对战（Local Duel）

### 1.1 基本开局与抽牌
1. 主菜单选择 **本地对战**。
2. 任选两名玩家、任选两套合法卡组（40 张）。
3. 进入对战后点击 **开始对战**。

**预期结果**：
- 双方玩家 HP 均为 30。
- 双方手牌数量一致（通常为 5 张）。
- 棋盘 5×5 正常渲染，列名显示为：高地、山脊、中路、河岸、水路。
- 无异常弹窗或 Traceback。

### 1.2 普通异象部署
1. 选择一张费用仅含 `T` 的异象卡（如 `书架`）。
2. 点击合法友方位置（p1 为第 3、4 行；p2 为第 0、1 行）。
3. 确认部署。

**预期结果**：
- T 点正确扣除。
- 异象出现在目标格，头像、攻击力/生命值显示正确。
- 手牌中该卡消失。

### 1.3 献祭部署（重点）
1. 确保场上至少有一只友方异象（如 `松鼠`）。
2. 选择一张需要 `B` 点费用的冥刻异象（如 `林鼠`，费用 `2T1B`）。
3. 在弹出的献祭选择中，选中场上友方异象。
4. 确认部署。

**预期结果**：
- 被选中的献祭异象 **立即从战场移除**，而不是变成 HP 0 继续存在。
- 玩家 B 点正确增加（`丰饶` 关键词按数值计算，默认 1）。
- 新异象正常部署到目标位置。
- 若献祭的是 `松鼠` 等 token，token 应被移除，不触发亡语（token 无亡语时）。
- **反向验证**：故意选择 B 点不足时，应提示“献祭不足”或无法完成出牌。

### 1.4 多祭品与双倍血契
1. 通过 `含垢齿轮` 或相关效果获得 `_double_blood_gain`。
2. 选择需要多只祭品的异象，或一次性选中多只异象献祭。

**预期结果**：
- 每只祭品都被移除。
- 获得的 B 点翻倍。
- 部署后 B 点不溢出、不回滚异常。

### 1.5 攻击与战斗结算
1. 回合进入 **结算阶段**（resolve）。
2. 观察场上异象互相攻击。
3. 手动设置攻击目标（若支持）。

**预期结果**：
- 攻击造成正确伤害。
- HP ≤ 0 的异象被移除。
- 战斗伤害触发相关关键词（如 `尖刺`、`亡语`、`空袭` 等）。
- 玩家直接攻击时，玩家 HP 正确减少。

### 1.6 拍铃与拉闸
1. 在行动阶段点击 **拍铃**（若 UI 提供）。
2. 连续让双方拉闸。

**预期结果**：
- 拍铃后本回合未出牌则失去 1 T 点。
- 双方拉闸后行动阶段结束，进入结算阶段。

## 2. 联机对战（Network Duel）

### 2.1 Host 创建与 Client 加入
1. 主机：主菜单选择 **创建房间**，记录 IP / 端口号 / ngrok 公网地址。
2. 客机：选择 **加入房间**，输入主机地址。

**预期结果**：
- 双方握手成功，进入对战准备界面。
- 客机能看到主机玩家名称。
- 双方卡组校验通过（非法卡组应被拒绝并提示）。

### 2.2 卡组合法性校验
1. 主机使用合法 40 张卡组。
2. 客机使用牌数不足 40、或沉浸点与卡牌包不匹配、或携带未知卡牌的卡组。

**预期结果**：
- 合法卡组正常进入对战。
- 非法卡组在握手阶段被拒绝，并显示具体原因。

### 2.3 行动同步
1. Host 部署一只异象。
2. Client 观察 Host 的部署是否同步显示。
3. Client 使用一张策略卡指向 Host 的异象。
4. Host 观察效果是否同步。

**预期结果**：
- 双方棋盘状态一致。
- 无错位、无延迟导致的“重影”或“HP 不同步”。

### 2.4 献祭同步（重点）
1. Host 在场上放置友方异象。
2. Host 打出需要 B 点的冥刻异象，并选择友方异象献祭。
3. Client 观察 Host 的棋盘。

**预期结果**：
- Host 与 Client 同时看到献祭异象被移除。
- 不会有一方看到 HP 0 残留。

### 2.5 心跳与断线
1. 正常联机对战 1-2 回合。
2. 模拟网络短暂卡顿（如关闭客机网络 3-5 秒后恢复）。
3. 观察连接状态。

**预期结果**：
- 短暂卡顿后，对战继续，无 5 秒超时强制默认操作。
- 心跳保持连接活跃，不会误报断开。
- 若一方真正断开，另一方应收到 `DISCONNECT` 提示，并安全结束对战。

### 2.6 异常消息校验
1. 尝试修改客户端发送非法 `ACTION`（如费用不足、非法目标、不存在的卡牌）。

**预期结果**：
- 服务端校验拒绝非法行动，对战状态不损坏。
- 不会崩溃或进入死锁。

## 3. 核心机制专项测试

### 3.1 地形与部署限制
| 测试项 | 步骤 | 预期 |
|--------|------|------|
| 水路限制 | 尝试把非水生/两栖异象部署到第 4 列（水路） | 部署失败 |
| 高地限制 | 尝试把带 `高地` 关键词的异象（如 `兀鹫`）部署到非第 0 列 | 部署失败 |
| 河岸限制 | 尝试把带 `河岸` 关键词的异象（如 `钓鱼竿`）部署到非第 3 列 | 部署失败 |
| 友方行限制 | p1 尝试在第 0/1 行部署，p2 尝试在第 3/4 行部署 | 部署失败 |
| 协同/独行 | 在已有同列友方异象的列部署新异象 | 遵循独行/协同规则 |

### 3.2 指向与目标
1. 使用需要指向目标的策略卡（如 `劫掠`）。
2. 点击合法/非法目标。

**预期结果**：
- 合法目标高亮，非法目标不响应或提示“无合法目标”。
- 指向取消后卡牌回到手牌，费用不扣除。

### 3.3 回合阶段
1. 观察每个回合的 **回合开始 → 行动阶段 → 结算阶段 → 回合结束**。

**预期结果**：
- 阶段切换正确。
- 回合开始/结束触发的效果正确执行。
- 结算阶段异象按规则攻击。

### 3.4 卡组构筑与沉浸点
1. 进入卡组编辑器。
2. 尝试保存以下卡组：
   - 40 张，沉浸点分配合理。
   - 39 张或 41 张。
   - 某非通用卡包未分配沉浸点但包含该包卡牌。
   - 某卡包分配 1 点沉浸点但只带 5 张该包卡（不足 10 张）。

**预期结果**：
- 合法卡组保存成功。
- 非法卡组给出明确错误提示。

## 6. 已知高风险点（每次发布前必须重点验证）

- [ ] **献祭后祭品残留**：选中献祭的异象应被移除，而不是 HP 降低为 0 后仍留在场上。
- [ ] **联机行动不同步**：Host/Client 双方棋盘状态应完全一致。
- [ ] **心跳误断开**：短暂卡顿不应导致对局结束。
- [ ] **非法行动破坏状态**：费用不足、目标非法、卡组非法等情况应被安全拒绝。
- [ ] **UI 启动报错**：`typing` 导入、`List`/`Tuple` 等类型注解错误不应再出现。

## 附录：快速排障

- **启动报 `NameError: name 'List' is not defined`**：检查 `gui/battle/board_renderer.py` 的 `typing` 导入是否包含 `List` 和 `Tuple`。
- **献祭后祭品 HP 0 不消失**：检查 `tards/cards.py` `_default_minion_effect` 中是否正确调用 `m.minion_death()`，并确认 `effect_queue` 后续处理了移除队列。
- **联机握手失败**：检查双方卡组是否 40 张、沉浸点是否合理；查看控制台 `NetworkDuel._validate_deck_list` 输出。
- **测试失败**：先单独运行失败用例：`python tests/run.py <test_name>`，确认是测试数据过期还是引擎 bug。
